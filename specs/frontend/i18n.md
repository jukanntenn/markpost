# Frontend Internationalization

## Overview

The frontend uses [next-intl](https://next-intl.dev/) for internationalization. It supports English (`en`) and Chinese (`zh`), with English as the default.

## Configuration

### Locale Constants

`src/i18n/constants.ts` defines available locales:

```typescript
export const availableLocales = ["en", "zh"] as const;
export type Locale = (typeof availableLocales)[number];
```

### Request Config

`src/i18n/request.ts` provides the server-side locale resolution:

```typescript
export default getRequestConfig(async () => {
  let locale: Locale = defaultLocale;
  const cookieStore = await cookies();
  const stored = cookieStore.get("locale")?.value;
  if (stored && availableLocales.includes(stored as Locale)) {
    locale = stored as Locale;
  }
  return {
    locale,
    messages: (await import(`./locales/${locale}.json`)).default,
  };
});
```

Locale is resolved by checking the `locale` cookie. If not set or invalid, defaults to English.

## Locale Files

```
src/i18n/locales/
├── en.json    — English translations
└── zh.json    — Chinese translations
```

Locale files are JSON objects with nested keys:

```json
{
  "dashboard": {
    "postKey": {
      "title": "Post Key",
      "description": "Use this key to..."
    }
  }
}
```

## Translation Key Naming

Keys use dot-separated, feature-prefixed names:

- `dashboard.postKey.title` — Dashboard feature, post key section
- `auth.login.title` — Auth feature, login page
- `common.save` — Shared translations

This mirrors the route structure and component organization.

## Usage in Components

Use the `useTranslations()` hook from next-intl:

```typescript
import { useTranslations } from "next-intl";

function PostKeySection() {
  const t = useTranslations("dashboard.postKey");
  return <h2>{t("title")}</h2>;
}
```

The namespace passed to `useTranslations()` selects the sub-tree from the locale file.

## LocaleProvider

The root layout wraps the app in `LocaleProvider` which receives the server-resolved locale and messages as props. Client components access translations through the next-intl context.
