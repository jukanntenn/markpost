# Backend Internationalization

## 技术栈

后端使用 `gin-contrib/i18n`（v1.2.2）+ `nicksnyder/go-i18n/v2`（v2.6.0）做服务端消息本地化。错误消息和响应字符串根据客户端的 `Accept-Language` 头翻译。locale 文件为 TOML 格式，存放在 `backend/locales/`。

## locale 文件组织

```
backend/locales/
├── active.en.toml         ← 英文（sourceLanguage，goi18n extract 产物，保留）
├── active.zh-Hans.toml    ← 简体中文
├── active.zh-Hant.toml    ← 繁体中文（台湾）
├── active.ja.toml         ← 日文
└── translate.*.toml       ← 临时待翻译文件（不入 git，翻译完成后删除）
```

- **BCP 47 命名**：locale 标签遵循 BCP 47（`en`、`zh-Hans`、`zh-Hant`、`ja`），匹配 `Accept-Language` 头。
- **默认语言 `en`**：缺失翻译时兜底英文。
- **支持语言**：en / zh-Hans / zh-Hant / ja，预留扩展（未来加新语言只需加 `active.<locale>.toml` 文件）。
- **`active.en.toml` 保留**：作为 goi18n extract/merge 工作流的基准文件（见下方"翻译工作流"）。
- **`translate.*.toml` 不入 git**：翻译完成即删，`active.*.toml` 才是真源。

## message 格式（模板插值）

TOML 格式，每个 entry 是一个 TOML table，用 `other` 字段存翻译字符串：

```toml
["error.validation_min_length"]
other = "{{.Field}} must be at least {{.Min}} characters"

["post.created"]
one = "{{.Count}} post created"
other = "{{.Count}} posts created"
```

- **模板插值**：用 Go [text/template](https://pkg.go.dev/text/template) 语法 `{{.Key}}`，通过 `LocalizeConfig.TemplateData` 传值。
- **复数**：用 CLDR plural forms（`one`/`other`/`few`/`many` 等），通过 `LocalizeConfig.PluralCount` 选择。

## message ID 命名规范

全小写下划线，feature 前缀分组：

| 类别 | 命名格式 | 例子 |
|---|---|---|
| 错误消息 | `error.<code>` | `error.invalid_credentials`、`error.not_found` |
| 字段校验消息 | `error.validation_<tag>` | `error.validation_required`、`error.validation_min_length` |
| 成功/提示消息 | `<feature>.<action>` | `auth.logout_success`、`post.created` |
| 健康检查 | `health.<key>` | `health.running` |

**成功消息不用 `error.` 前缀**（如 `auth.logout_success`，不是 `error.logout_success`）。

## 双源结构

message 有两个来源，运行时协同工作：

| 源 | 内容 | 角色 |
|---|---|---|
| **代码内 `ErrCode.Message`** | 英文 `DefaultMessage`（含 ID + Other） | 权威兜底来源 |
| **locale 文件（active.*.toml）** | 各语言翻译 | 提供非英文翻译 |

### 运行时渲染（apierr）

`apierr` 用 `ginI18n.MustGetMessage` 渲染，`DefaultMessage` 指向 ErrCode 自带的 Message：

```go
func renderMessage(c *gin.Context, code *service.ErrCode, data map[string]any) string {
    msg := ginI18n.MustGetMessage(c, &i18n.LocalizeConfig{
        DefaultMessage: code.Message,   // 代码内英文兜底
        TemplateData:   data,           // {{.Field}} {{.Min}} 等
    })
    if msg == "" {
        return code.Message.Other       // 终极兜底：英文原文
    }
    return msg
}
```

**渲染逻辑**（go-i18n 的 `Localize` 四级兜底）：
1. 先在请求语言（如 zh-Hans）的 locale 文件找 → 命中则用翻译
2. 找不到 → fallback 到默认语言（en）的 locale 文件
3. 还找不到 → 用 `DefaultMessage`（代码内嵌英文）
4. DefaultMessage 为 nil → 返回空（被 apierr 的 `code.Message.Other` 终极兜底拦截）

**为什么保留 `active.en.toml`**：它和代码内的 DefaultMessage 内容重复（都是英文），但不冲突。它作为 goi18n extract/merge 工作流的基准文件存在，extract 产出它，merge 以它为基准生成其他语言的 translate 文件。

## message 字面量硬约束

### 规则：message 的所有字段必须是字符串字面量

goi18n extract 用 AST 扫描 Go 源码，提取 `i18n.Message` / `i18n.LocalizeConfig` 类型的 CompositeLit（复合字面量）。字段值只认：

- **字符串字面量**（`*ast.BasicLit`）：`"hello"` ✅
- **字符串拼接**（`*ast.BinaryExpr`）：`"a" + "b"` ✅
- **字符串常量引用**（`*ast.Ident` 指向 const 声明）：✅（但规范统一要求只写字面量，避免歧义）

**不认**：变量、函数返回值、非字符串常量。

### 正确写法

```go
var ErrInvalidCredentials = &ErrCode{
    Message: &i18n.Message{              // ← i18n.Message 字面量
        ID:    "error.invalid_credentials",
        Other: "Invalid username or password",  // ← 字面量，extract 能提取
    },
}
```

**关键**：ErrCode struct 里嵌套的 `&i18n.Message{...}` 本身就是字面量，goi18n extract 的 `ast.Walk` 会递归访问到它，**能正确提取**（extractor 扫描所有 `i18n.Message` 类型的 CompositeLit，不管外面包了什么 struct）。

### 错误写法

```go
// ❌ 变量，extract 提取不到
msg := "Invalid username or password"
var ErrInvalidCredentials = &ErrCode{
    Message: &i18n.Message{ID: "error.invalid_credentials", Other: msg},
}

// ❌ 函数返回值，extract 提取不到
Other: generateMessage("invalid_credentials"),
```

### 重复 ID 检测

goi18n extract 内置重复检测：如果两个 message 同 ID 但内容不同，extract 直接报错 `duplicate message ID: xxx`。这是内置防重保护，保证 message ID 全局唯一。

## 完整翻译工作流

### goi18n 命令安装

```bash
go install -v github.com/nicksnyder/go-i18n/v2/goi18n@latest
```

### 场景 1：新增 message（开发期最常见）

开发者在代码里写了新的 `&i18n.Message{ID:"post.created", Other:"Post created"}`：

```bash
# 1. 提取所有 message 到 active.en.toml
goi18n extract

# 2. 生成待翻译文件（含新增的 message）
goi18n merge active.*.toml

# 3. 翻译 translate.zh-Hans.toml / translate.zh-Hant.toml / translate.ja.toml

# 4. 合并翻译进 active.*.toml
goi18n merge active.*.toml translate.*.toml

# 5. 删除临时待翻译文件
rm translate.*.toml
```

### 场景 2：新增语言（如加韩文 ko）

```bash
# 1. 创建空文件
touch translate.ko.toml

# 2. 填充待翻译 message
goi18n merge active.en.toml translate.ko.toml

# 3. 翻译 translate.ko.toml

# 4. 重命名为 active
mv translate.ko.toml active.ko.toml

# 5. 在代码里注册加载
bundle.LoadMessageFile("active.ko.toml")
```

### 场景 3：修改已有 message 文案

改了英文原文（如 `Other` 从 "Invalid username or password" 改为 "Username or password is incorrect"）：

```bash
# 1. extract 更新 active.en.toml（该 message 的 hash 变了）
goi18n extract

# 2. merge 识别需重译项（hash 不匹配）
goi18n merge active.*.toml

# 3. 翻译变化的条目
# 4. 合并
goi18n merge active.*.toml translate.*.toml
```

### hash 字段机制

`goi18n merge` 生成的 `translate.*.toml` 每条 message 带 `hash = "sha1-..."`（message 内容的 sha1）。当英文原文改了，hash 变化，merge 能识别"这条需要重新翻译"。

### CI 集成建议

建议在 CI 加检查：`goi18n extract` 后若 `active.en.toml` 有 diff，说明有新 message 没走完整翻译流程，CI 报错提醒开发者完成翻译。

## 语言检测

语言从 `Accept-Language` HTTP 头检测（gin-contrib/i18n 的 `defaultGetLngHandler`）。如果头缺失或语言不支持，用默认语言 `en` 兜底。

## 代码用法

handler 用 `ginI18n.MustGetMessage` 获取翻译字符串（支持 TemplateData 插值）：

```go
message := ginI18n.MustGetMessage(c, &i18n.LocalizeConfig{
    DefaultMessage: &i18n.Message{
        ID:    "auth.logout_success",
        Other: "Logged out successfully",
    },
})
```

`DefaultMessage` 提供英文兜底。错误消息由 `apierr` 统一渲染（见 [error-handling.md](./error-handling.md) 的 apierr 包契约）。

## 配置

i18n 在 `cmd/server/main.go` 启动时配置：

```go
r.Use(ginI18n.Localize(ginI18n.WithBundle(&ginI18n.BundleCfg{
    RootPath:        "./locales",
    AcceptLanguage:  []language.Tag{language.English, language.Chinese, language.Japanese},
    DefaultLanguage: language.English,
    UnmarshalFunc:   toml.Unmarshal,
    FormatBundleFile: "toml",
})))
```
