# Dagger E2E Testing Wiki

This document captures the Dagger concepts and patterns used in Markpost's E2E test suite.

## Overview

[Dagger](https://dagger.io/) is a programmable CI/CD engine that runs pipelines as code inside containers. We use it to orchestrate full-stack E2E tests: building real services (Go backend, Next.js frontend, PostgreSQL) as containers, wiring them together, and running Playwright against them.

## Core Concepts

### Dagger Module

A Dagger module is a directory with a `dagger.json` config file containing functions decorated with `@object()` and `@func()`. Functions accept typed arguments (strings, `Directory`, `Container`, `Service`, etc.) and are callable via `dagger call`.

```json
// dagger.json
{
  "name": "markpost-e2e",
  "engineVersion": "v0.21.0",
  "sdk": { "source": "typescript" },
  "source": "src"
}
```

### TypeScript Module Structure

```
e2e/
├── dagger.json          # Module config
├── src/                 # Module source root (configured in dagger.json)
│   ├── package.json     # Runtime dependencies
│   ├── tsconfig.json    # SDK path mapping
│   ├── sdk/             # Bundled Dagger SDK (auto-generated by dagger develop)
│   └── src/
│       └── index.ts     # @object() class with @func() methods
├── tests/               # Playwright test files
├── lib/                 # Test support code
└── playwright.config.ts
```

The SDK is bundled locally under `sdk/` with path mapping in `tsconfig.json`:

```json
{
  "paths": {
    "@dagger.io/dagger": ["./sdk/index.ts"],
    "@dagger.io/dagger/telemetry": ["./sdk/telemetry.ts"]
  }
}
```

### Initialization Commands

```bash
cd e2e/
dagger init --name=markpost-e2e --sdk=typescript --source=src
dagger develop            # Regenerates SDK, sets up local dev environment
```

## Services Pattern

### Creating Services

Any container can be exposed as a network-accessible service using `.asService()`:

```typescript
const postgres = dag.container()
  .from("postgres:17-alpine")
  .withEnvVariable("POSTGRES_DB", "markpost")
  .withExposedPort(5432)
  .asService({ useEntrypoint: true })
```

### Binding Services

Services are connected to client containers via `withServiceBinding(alias, service)`. The alias becomes a hostname in the client's `/etc/hosts`:

```typescript
const backend = dag.container()
  .from("alpine:3.21")
  .withServiceBinding("postgres", postgres)  // backend resolves "postgres" hostname
  .withEnvVariable("MARKPOST_DB__DSN", "postgres://user:pass@postgres:5432/db")
  .asService({ args: ["/app/markpost", "serve"] })
```

### Service Lifecycle

- Services start **lazily** when a bound container runs
- Dagger **health-checks** services before running dependent containers
- Services are **de-duplicated** within a client session
- Services stop after a **10-second grace period** when no longer needed
- Use explicit `start()`/`stop()` for fine-grained control

### Interdependent Services

Use `withHostname()` for services that reference each other:

```typescript
const svcA = dag.container().from("nginx").withExposedPort(80).asService(...).withHostname("svca")
const svcB = dag.container().from("nginx").withExposedPort(80).asService(...).withHostname("svcb")
```

## Per-Test-File Isolation

Each test file gets its own complete sandbox of containers. This is achieved by creating fresh service instances inside each `test()` function call:

```typescript
@func()
async test(testFile: string, source: Directory): Promise<string> {
  const postgres = this.postgresService()          // Fresh postgres
  const backendSvc = this.backendService(..., postgres)   // Fresh backend
  const frontendSvc = this.frontendService(..., backendSvc) // Fresh frontend
  // Run Playwright against this isolated stack
}
```

Since services are content-addressed and scoped to a session, each `test()` invocation gets independent instances.

## Building Multi-Stage in Dagger

Dagger doesn't have Docker-style multi-stage builds. Instead, extract artifacts from one container and use them in another:

```typescript
// Build stage
const builder = dag.container()
  .from("golang:1.26-alpine")
  .withDirectory("/app", source)
  .withExec(["go", "build", "-o", "markpost", "./cmd/server"])

// Extract binary
const binary = builder.file("/app/markpost")

// Runtime stage
const runtime = dag.container()
  .from("alpine:3.21")
  .withFile("/app/markpost", binary)
```

For Next.js standalone builds, extract the standalone output, static files, and public directory:

```typescript
const standalone = builder.directory("/app/.next/standalone")
const staticFiles = builder.directory("/app/.next/static")
const publicFiles = builder.directory("/app/public")

const runtime = dag.container()
  .from("node:24-alpine")
  .withDirectory("/app", standalone)
  .withDirectory("/app/.next/static", staticFiles)
  .withDirectory("/app/public", publicFiles)
```

## Caching

Use `withMountedCache()` for dependency caches to speed up repeated builds:

```typescript
dag.container()
  .withMountedCache("/go/pkg/mod", dag.cacheVolume("go-mod-e2e"))
  .withMountedCache("/root/.cache/go-build", dag.cacheVolume("go-build-e2e"))
```

Cache volumes are persisted across Dagger runs. Use unique names to avoid conflicts between different pipelines.

## Passing External Source

Dagger modules are sandboxed and can't access the host filesystem directly. Pass project source as a `Directory` argument:

```bash
# From project root
dagger call -m e2e test --test-file=login.spec.ts --source=..
dagger call -m e2e all --source=..
```

Inside the module, access subdirectories:

```typescript
source.directory("backend")   // Backend source
source.directory("frontend")  // Frontend source
source.directory("e2e/tests") // Test files
```

## "All" Function Pattern

Run test files **sequentially** (not in parallel) to avoid resource contention and state leakage between Dagger sandboxes:

```typescript
@func()
async all(source: Directory): Promise<string> {
  const specs = await source.directory("e2e/tests").entries()
  const results: string[] = []
  for (const file of specs.filter(f => f.endsWith(".spec.ts"))) {
    try {
      const out = await this.test(file, source)
      results.push(`✅ ${file}\n${out}`)
    } catch (e) {
      results.push(`❌ ${file}\n${e instanceof Error ? e.message : String(e)}`)
    }
  }
  return results.join("\n\n" + "=".repeat(80) + "\n\n")
}
```

Key design decisions:
- **Sequential**: 6 sandboxes × 4 containers = 24 containers in parallel would overwhelm most machines. Sequential keeps peak resource usage at 4 containers.
- **Error isolation**: Each file's result is caught independently — one failure doesn't stop the others.
- **`ReturnType.Any`**: Use this on the Playwright `withExec` to capture stdout/stderr even on test failure.

Individual tests can still be run with `dagger call test --test-file=... --source=..`.

### Preventing State Leakage Across Sequential Runs

Dagger caches container layers. If two sequential `test()` calls use identical container definitions, Dagger may reuse the previous run's writable layer (e.g., a postgres database with mutated state). Prevent this by adding a unique `RUN_ID` env variable to each service:

```typescript
const runId = testFile + "-" + Date.now()
const postgres = this.postgresService(runId)
// Inside postgresService:
//   .withEnvVariable("RUN_ID", runId)
```

This forces Dagger to treat each run as a distinct container, guaranteeing a fresh database.

## Playwright in Dagger

Playwright runs inside a container with the official Docker image:

```typescript
dag.container()
  .from("mcr.microsoft.com/playwright:v1.53.0-noble")
  .withDirectory("/app/tests", testDir)
  .withDirectory("/app/lib", libDir)
  .withFile("/app/playwright.config.ts", config)
  .withFile("/app/package.json", packageJson)
  .withExec(["npm", "install"])
  .withServiceBinding("frontend", frontendSvc)
  .withServiceBinding("backend", backendSvc)
  .withEnvVariable("FRONTEND_URL", "http://frontend:3000")
  .withEnvVariable("BACKEND_URL", "http://backend:7330")
  .withExec(["npx", "playwright", "test", "--config=playwright.config.ts", testFile])
```

Key points:
- Browser processes share the container's `/etc/hosts`, so service hostnames are resolvable
- `FRONTEND_URL` and `BACKEND_URL` env vars tell the test code where services are
- The Playwright config uses `FRONTEND_URL` as `baseURL`
- CORS must allow cross-origin requests (the backend defaults to `AllowOrigins: ["*"]`)

## Docker Hub Rate Limits

Dagger resolves container images independently from the local Docker daemon. Each `dagger call` session resolves image manifests from Docker Hub, counting against the unauthenticated rate limit (100 pulls/6hr/IP). Mitigation:

1. **Authenticate**: Run `docker login`. Dagger reads credentials from `~/.docker/config.json` (via `config.LoadDefaultConfigFile` in the Go client). Free accounts get 200 pulls/6hr.
2. **Minimize sessions**: Use `dagger call all` (one session) instead of multiple `dagger call test` invocations.
3. **Registry mirrors**: Configure `registries` in the Dagger engine config to use a mirror.

## Capturing Test Output on Failure

By default, `withExec` throws on non-zero exit codes, discarding stdout. To capture Playwright output even when tests fail:

```typescript
import { ReturnType } from "@dagger.io/dagger"

const runner = container
  .withExec(["npx", "playwright", "test", ...], { expect: ReturnType.Any })

const exitCode = await runner.exitCode()
const stdout = await runner.stdout()
if (exitCode !== 0) {
  throw new Error(stdout)
}
return stdout
```

`ReturnType.Any` allows exit codes 0-127 and 192-255 without throwing. Then check `exitCode()` manually.

## Troubleshooting

| Issue | Solution |
|---|---|
| `dagger` not found | Install: `curl -L https://dl.dagger.io/dagger/install.sh \| sh` |
| Module code changes not taking effect | Run `dagger develop` to regenerate SDK |
| Services not connecting | Verify `withServiceBinding` alias matches the hostname used in config |
| Build cache stale | Use unique cache volume names or clear with `docker system prune` |
| `NEXT_PUBLIC_*` vars wrong | These are baked at build time; rebuild frontend when changing them |
| Next.js `role="alert"` false positive | Next.js adds a `__next-route-announcer__` element with `role="alert"`. Use component-specific selectors (e.g., `[data-slot='alert']`) instead of `getByRole("alert")`. |
| 401 on sequential test runs | Dagger may reuse cached service containers. Add unique `RUN_ID` env var to each service. |
| Go binary won't run on Alpine | Use `CGO_ENABLED=1` and run in the golang build container directly instead of extracting the binary. |
