# Frontend Internationalization

本文档定义前端 i18n 架构。后端 i18n 见 [backend/i18n.md](../backend/i18n.md)；前后端联动见本文档末尾。

## 技术栈

前端使用 [next-intl](https://next-intl.dev/)（v4.7.0）做客户端消息本地化。**纯客户端模式**——不使用 `getRequestConfig` / `next-intl/plugin` / `i18n/request.ts`（这些是服务端装配，纯静态导出不可用，详见 [build.md](./build.md) §3）。

> 依据（next-intl 源码）：`NextIntlClientProvider` 是 `'use client'` 组件（`NextIntlClientProvider.tsx:1`），接收 `locale` + `messages` props 即可工作，不依赖服务端。`getRequestConfig` / `i18n/request.ts` 是"server-only"（`configuration.mdx:16`）。纯静态导出跳过这些，直接用 provider 传 props。

## 支持语言

与后端对齐，支持 4 种语言（BCP 47 标签）：

| 标签 | 语言 | 默认 |
|------|------|------|
| `en` | English | ✅ |
| `zh-Hans` | 简体中文 | |
| `zh-Hant` | 繁体中文（台湾） | |
| `ja` | 日本語 | |

```typescript
// src/i18n/constants.ts
export const availableLocales = ["en", "zh-Hans", "zh-Hant", "ja"] as const;
export type Locale = (typeof availableLocales)[number];
export const defaultLocale: Locale = "en";
```

> **BCP 47 对齐**：前端用 `zh-Hans` / `zh-Hant`（区分简繁），与后端 locale 标签一致。发请求时 `Accept-Language` 直接传当前 locale，后端能精确匹配。不使用不区分简繁的 `zh`。

## Locale 文件

```
src/i18n/locales/
├── en.json         English（基准）
├── zh-Hans.json    简体中文
├── zh-Hant.json    繁体中文
└── ja.json         日文
```

每语言一个 JSON 文件，嵌套对象按 namespace 分组：

```json
{
  "dashboard": {
    "postKey": {
      "title": "Post Key",
      "description": "Use this key to..."
    }
  },
  "common": {
    "save": "Save",
    "cancel": "Cancel"
  }
}
```

### 翻译 key 命名

dot-separated、feature 前缀，与后端消息 ID 风格一致：

- `dashboard.postKey.title` — dashboard 功能，post key 区
- `login.username` — 登录页
- `common.save` — 共享翻译

namespace 映射路由结构和组件组织。

## 语言检测与持久化

### 检测优先级

```typescript
// src/utils/i18n.ts
export function getDefaultLocale(): Locale {
  if (typeof window === "undefined") return "en";
  // 1. localStorage
  const stored = localStorage.getItem("locale");
  if (stored && availableLocales.includes(stored as Locale)) return stored as Locale;
  // 2. 浏览器语言
  const browserLang = navigator.language.split("-")[0];
  if (availableLocales.includes(browserLang as Locale)) return browserLang as Locale;
  // 3. 回退 en
  return "en";
}
```

### 持久化

```typescript
export function persistLocale(locale: Locale): void {
  localStorage.setItem("locale", locale);
}
```

**只存 localStorage**，不写 cookie（后端用 `Accept-Language` 头检测，不用 cookie）。

## 动态加载 messages

locale 切换时动态 import 对应 JSON 文件（构建时打成独立 chunk，按需加载）：

```typescript
export async function loadMessages(locale: Locale) {
  return (await import(`@/i18n/locales/${locale}.json`)).default;
}
```

## Provider 装配（纯客户端自举）

`LocaleProvider` 是 `'use client'` 组件，用 `NextIntlClientProvider` 包裹应用。**不接收 serverLocale / serverMessages props**（root layout 不调服务端 i18n API）。

```tsx
// src/components/providers/LocaleProvider.tsx
"use client";

export function LocaleProvider({ children }: { children: React.ReactNode }) {
  const [locale, setLocaleState] = useState<Locale>("en");
  const [messages, setMessages] = useState<AbstractIntlMessages>({});

  // hydration 后从 localStorage 读取真实 locale + 加载 messages
  useEffect(() => {
    const stored = getDefaultLocale();
    loadMessages(stored).then((m) => {
      setLocaleState(stored);
      setMessages(m);
    });
  }, []);

  const setLocale = useCallback(async (newLocale: Locale) => {
    const m = await loadMessages(newLocale);
    setLocaleState(newLocale);
    setMessages(m);
    persistLocale(newLocale);
    document.documentElement.lang = newLocale;
  }, []);

  return (
    <LocaleContext.Provider value={{ locale, setLocale, availableLocales }}>
      <NextIntlClientProvider locale={locale} messages={messages}>
        {children}
      </NextIntlClientProvider>
    </LocaleContext.Provider>
  );
}
```

**首次渲染 `messages={}`**（只显示中性 Loading 骨架，不渲染任何语言的 UI 文案），hydration 后 `useEffect` 异步加载用户偏好 locale 的 messages chunk 并重渲染。这样避免了「先英文后切换」的闪烁——非 en 用户不会看到英文一闪而过。

代价：next-intl 在 `messages={}` 那一帧会对每个被访问的 namespace 抛 `MISSING_MESSAGE`。`LocaleProvider` 的 `onError` 在「messages 尚未加载」窗口内静默这些错误（纯噪音），加载完成后真正的缺 key 错误仍正常上报。

> root layout 不调用 `getLocale()` / `getMessages()`（这些是 `next-intl/server` API，纯静态导出不可用）。详见 [architecture.md](./architecture.md) Provider Stack。

## 在组件中使用

用 `useTranslations(namespace?)` hook（next-intl 提供）：

```typescript
import { useTranslations } from "next-intl";

function PostKeySection() {
  const t = useTranslations("dashboard.postKey");
  return <h2>{t("title")}</h2>;
}
```

`useTranslations` 的 namespace 参数可选——不传则返回所有 messages。namespace 支持点号嵌套（`useTranslations("dashboard.postKey")`）。

`useLocale()` 返回当前 locale（来自 provider 的 `locale` prop）。

## 翻译工作流

前端 locale 文件**手动维护**，`en.json` 为基准。条目数不多（现状 ~273），手动维护足够。

新增翻译 key 时：在 `en.json` 加条目 → 翻译到 `zh-Hans.json` / `zh-Hant.json` / `ja.json`。

> 后端用 goi18n extract/merge 工作流（见 [backend/i18n.md](../backend/i18n.md)），前端不引入提取工具（next-intl 不绑定特定提取工具，引入额外复杂度无收益）。

## 前后端联动

| 维度 | 规范 |
|------|------|
| 语言范围 | 前后端一致：en / zh-Hans / zh-Hant / ja |
| 标签 | 前后端一致：BCP 47 |
| 语言传递 | 前端 API client 每请求带 `Accept-Language: <当前 locale>`；后端按此头翻译错误消息 |
| 消息 ID 风格 | 前后端一致：dot-separated feature 前缀（前端 `login.username`，后端 `error.invalid_credentials`） |

前端选语言 → `persistLocale` 存 localStorage → API client 读当前 locale → 每请求带 `Accept-Language` 头 → 后端返回对应语言的错误消息。
